# 01 — Vision & Scope: อู้ฟู่ (Oofoo)

> สถานะ: ร่างแรก (Phase 1) — ส่วนที่มี ❓ คือการตัดสินใจที่รอเคาะ

## 1. Vision Statement
แอปสำหรับสั่งซื้อสินค้าอุปโภคบริโภคจากร้านของชำชุมชน **"อู้ฟู่"** ที่ให้ลูกค้าเลือกได้ระหว่าง
**เดลิเวอรี่ส่งถึงบ้าน** กับ **สั่งออนไลน์รับที่ร้าน** — ใช้ง่าย จ่ายเงินแบบไทย (PromptPay / ปลายทาง) เห็นของสดใหม่ในราคาที่จับต้องได้

## 2. Problem / Opportunity
ร้านของชำชุมชนแข่งกับร้านสะดวกซื้อเชนใหญ่ได้ยากเพราะไม่มีช่องทางออนไลน์ ลูกค้าในละแวกอยากได้ความสะดวก
"สั่งแล้วได้ของ" โดยไม่ต้องเดินทาง และคุ้นกับการจ่าย PromptPay/เก็บปลายทาง อู้ฟู่เปิดช่องให้ร้านเล็กมีหน้าร้านดิจิทัล
ที่รองรับทั้งส่งถึงบ้านและรับเองที่ร้าน

## 3. Goals & Success Metrics (KPI)
| ด้าน | เป้าหมาย | ตัววัด |
|------|---------|--------|
| Business | เพิ่มยอดขาย/ออเดอร์ต่อวัน | จำนวนออเดอร์/วัน, AOV (มูลค่าต่อตะกร้า) |
| Product | ซื้อซ้ำ + ใช้ง่าย | retention 30 วัน, conversion (เปิดแอป→สั่งสำเร็จ), เวลาเฉลี่ยจนสั่งเสร็จ |
| Ops | จัดการออเดอร์ทัน | เวลาตั้งแต่รับออเดอร์→ส่ง/พร้อมรับ |
| Quality | ใช้ได้กับทุกคน + เสถียร | crash-free sessions ≥ 99.5%, ผ่าน WCAG AA, รองรับฟอนต์ใหญ่ |

## 4. Target Market & Users
ผู้บริโภคไทย mobile-first ในละแวกร้าน (รายละเอียดใน `02-personas.md`): ลูกค้าวัยทำงาน, ลูกค้าสูงวัย, และฝั่งร้าน (เจ้าของ/แอดมิน)

## 5. Scope

ระบบ v1 มี **3 surfaces** (ร้านเดียว อู้ฟู่ — ไม่ใช่ marketplace):

**(ก) แอปลูกค้า (Expo iOS/Android)**
- ดูสินค้า (หน้าหลัก/หมวดหมู่), ค้นหา, หน้ารายละเอียดสินค้า
- ตะกร้า, รายการโปรด (wishlist)
- **2 โหมด**: เดลิเวอรี่ (มีค่าส่ง/ส่งฟรีเมื่อถึงขั้นต่ำ) และ ออนไลน์/รับที่ร้าน (pickup)
- เข้าสู่ระบบ: **เบอร์ + OTP และ LINE/Apple/Google** (+ PDPA consent)
- Checkout → PromptPay QR + แนบสลิป → ยืนยันออเดอร์
- ติดตามสถานะออเดอร์ + ประวัติ + สั่งซ้ำ, การแจ้งเตือน (push)

**(ข) เว็บแอดมิน (เจ้าของร้าน/พนักงาน)**
- จัดการแคตตาล็อก (สินค้า/หมวด/ราคา/สต็อก/รูป)
- จัดการออเดอร์ + เปลี่ยนสถานะ + มอบหมายไรเดอร์
- ตรวจสลิปชำระเงิน (อนุมัติ/ปฏิเสธ)
- แดชบอร์ด KPI (lean สำหรับ v1)

**(ค) แอปไรเดอร์ (ผู้ส่งของร้าน)**
- เข้าสู่ระบบ + สถานะออนไลน์/ออฟไลน์
- คิวงานส่ง + รับงาน
- งานส่งที่กำลังทำ: นำทาง/ติดต่อลูกค้า + หลักฐานการส่ง (proof of delivery)

### 5.2 Out of Scope (v1) / อนาคต
- โปรแกรมสะสมแต้ม/คูปองขั้นสูง (v1 มีช่องโค้ดแบบพื้นฐาน)
- รีวิว/เรตติ้งจากผู้ใช้จริง (v1 ใช้เรตติ้งจากแคตตาล็อก)
- แชตกับร้าน, การคืนเงิน/คืนสินค้าแบบอัตโนมัติ
- เว็บ (โฟกัส iOS/Android ก่อน)

### 5.3 Scope Decisions — สรุปแล้ว ✅ (2026-06-27)
1. ✅ **รูปแบบร้าน**: ร้านเดียว (อู้ฟู่) — ออกแบบ data model เผื่อขยายหลายร้านได้ แต่ไม่อยู่ใน v1
2. ✅ **เข้าสู่ระบบ**: เบอร์ + OTP **และ** LINE / Apple / Google
3. ✅ **ขอบเขต**: แอปลูกค้า + เว็บแอดมิน + แอปไรเดอร์ (เดลิเวอรี่ของร้านเอง)
4. 🟠 **การชำระเงิน (เปิดอยู่ — ADR-pending):** v1 ใช้ **PromptPay + แนบสลิป (แอดมินตรวจมือ)** —
   ยังต้องคุยกันต่อเรื่อง: (ก) จะใช้ payment gateway อัตโนมัติ (Omise/2C2P/GB Prime) ไหม,
   (ข) รองรับ COD ไหม, (ค) **โหมดเดลิเวอรี่จ่ายก่อน หรือเก็บปลายทาง** — ข้อนี้กระทบ flow ไรเดอร์โดยตรง

## 6. Assumptions
- ลูกค้าเป้าหมายมีสมาร์ตโฟนและใช้ PromptPay เป็น
- ร้านมีคนคอยจัดการออเดอร์ในเวลาทำการ
- สินค้าเริ่มต้นเป็นของชำ/ของสด (ตามแคตตาล็อกตัวอย่างปัจจุบัน)

## 7. Constraints
- เทคโนโลยี: Expo SDK 54, React Native 0.81, expo-router 6 (ตามฐานปัจจุบัน)
- ภาษา: ไทยเป็นหลัก (รองรับ i18n เผื่ออนาคต)
- กฎหมาย: ต้องสอดคล้อง **PDPA** (พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล) — เก็บ/ใช้ข้อมูลลูกค้าอย่างถูกต้อง
- งบ/ทีม: ตามที่ Product Owner กำหนด (TBD)

## 8. Risks (เบื้องต้น)
- การตรวจสลิปด้วยมือไม่สเกล หากออเดอร์เยอะ → พิจารณา payment gateway
- ความถูกต้องของสต็อก/ราคาเรียลไทม์ หากไม่มีระบบหลังบ้านจัดการ
- หนี้ทางเทคนิคจากฐานปัจจุบัน (ดู `00-current-state-assessment.md`) ต้องสะสางก่อนต่อยอด

## 9. Non-Functional Targets (สรุประดับสูง — รายละเอียดใน `04-`)
Accessibility (WCAG AA), Performance (เปิดเร็ว/ลิสต์ลื่น), Security & Privacy (PDPA, auth ปลอดภัย),
Reliability (crash-free ≥ 99.5%), Observability (crash/analytics), i18n-ready
